Serverless Framework は AWS Lambda をはじめとするサーバーレス環境のデプロイを支援するツールとして、長年多くの開発者に利用されてきました。

https://github.com/serverless/serverless/

しかし、v4 系のリリース以降、OSS としての透明性に関する問題が指摘されていました。
この記事では、Serverless Framework v4 で何が起こっていたのか、そして 2024 年 12 月のコミットで何が変わったのかを整理したいと思います。

## 何が起こっていたのか

2024 年初頭から、Serverless Framework のリポジトリで通常とは異なる状態が続いていました。

- v4 系のリリースは継続的に公開されていた
- しかし、GitHub 上のソースコードは長期間更新されていなかった
- リリースノートに記載された変更に対応するコードが見当たらない状態が続いた
- `CONTRIBUTING.md` に従ってローカルでテストを実行しようとしても、正しく動作しないになっていた

この状態は、[Issue #12761](https://github.com/serverless/serverless/issues/12761) で指摘されています。利用者からすると、「何がリリースされているのかわからない」「コードを検証できない」という状況でした。

https://github.com/serverless/serverless/issues/12761

## OSS における透明性とは

OSS の価値は、単に「無償で使える」ことではありません。利用者が以下のことを行えることが、OSS の前提条件です。

- ソースコードを閲覧し、動作を検証できる
- 問題を発見したら報告・修正できる
- 修正内容をコミュニティで共有できる

この「検証可能性」と「参加可能性」が失われると、たとえライセンスが OSS であっても、実質的にはブラックボックスな配布物と変わりません。

参考：[オープンソースの定義 | Open Source Group Japan](https://opensource.jp/osd/osd17/)

## 外部貢献が困難だった具体例

透明性の欠如は、コミュニティからの貢献を阻害します。
ここでは、筆者自身が経験した具体例を紹介します。

筆者は、Windows 環境でパス区切りが混在した場合に正しく動作しない箇所について、
`path.basename()` へ置き換える修正を PR として提案しました。

https://github.com/serverless/serverless/pull/13119

修正内容自体は小さく、影響範囲も限定的なものでしたが、
この PR はマージされずに Close されています。

当時、筆者が直面した状況は次のようなものでした。

- 公開されているソースコードでは、ローカルでテストを正しく実行できない
- CI の導線が弱く、変更内容を十分に検証できない
- 実際に配布されている成果物との差分を確認できない

そのため、「修正内容が正しいか」を貢献者自身で十分に担保することが難しい状態でした。

重要なのは、この件が「PR が受け入れられなかった」という話ではない点です。  
**貢献しようとしても、検証可能な環境が整っていなかった** という構造的な問題を示しています。

## OSS のスケーラビリティと貢献の重要性

OSS は「利用者 = 潜在的貢献者」という構造によってスケールします。

貢献できない状態が続くと：

- すべての修正がメンテナに集中する
- 小さなバグや改善も滞留しやすくなる
- プロジェクト規模が大きいほど、持続性が低下する

これは Serverless Framework 固有の問題ではなく、OSS 全般に当てはまる原則です。

## 12 月のコミットで何が変わったか

2024 年 12 月、[`8cec971f`](https://github.com/serverless/serverless/commit/8cec971f1dd815b6ef8ab920acf35837851fb4ac) で、v4 系の実態に対応するソースコードが GitHub に公開されました。

https://github.com/serverless/serverless/commit/8cec971f1dd815b6ef8ab920acf35837851fb4ac

これにより、

- ソースとリリースの対応関係が回復した
- 利用者が変更内容を確認できるようになった
- 最悪の状態（ブラックボックスな配布）からは脱した

**「最低限の透明性」は戻った**と言えるでしょう。

## 現状の評価

現時点での状況を整理します。

**改善した点：**

- ソースコードとリリースノートの対応が取れるようになった
- 利用者が変更内容を検証できる状態に戻った

**今後の課題：**

- PR や Issue が今後どのように扱われるかは、まだ確認段階
- 一度失われた信頼は、行動の積み重ねでしか回復しない

透明性は回復しましたが、健全な OSS 運営に戻ったかは、今後の運用次第です。

## 有償化との関係

Serverless Framework v4 では、ライセンス／価格モデルが変更されています。
Serverless 側の説明では、年商が一定額（例：$2M）を超える組織に対して v4 利用に有料サブスクリプションを求めるモデルです。  
また v4 ではログインが必須になるなど、従来よりも Dashboard と結びついた運用になっています。

ここで明確にしておきたいのは、**有償化（商用モデル化）自体が悪いのではない**という点で、問題は以下です。

- リリースとソースの対応が取れず、利用者が検証できない状態が続いたこと
- OSS とプロプライエタリの境界が利用者視点で分かりづらかったこと

今後は、その線引きがどのように示され、継続的に守られるかを見ていきたいです。

## まとめ

Serverless Framework v4 は、一度失われていた OSS としての最低限の透明性は回復しました。しかし、健全な OSS 運営に戻ったかは、今後の運用次第です。

この記事で扱ったのは、

- 何が起きたか（事実の整理）
- OSS にとって透明性がなぜ重要か（原則の確認）
- 今後何を見るべきか（判断材料の提示）

Serverless Inc. の内部事情や意図については、推測の域を出ないため触れていません。重要なのは、**利用者として何を基準に判断すべきか**です。

今後、PR や Issue がどう扱われるか、コミュニティとの対話がどう行われるか。それらの積み重ねが、真に OSS としての信頼を取り戻せるかを決めるでしょう。
